# 学習済みモデルについて

このプロジェクトの学習部分は、強さそのものを極限まで追うための大規模学習ではありません。就活ポートフォリオとして、次の 3 点を説明しやすくするための軽量な自己対局ベースです。

同梱モデル `selfplay-v2-xl` は、自己対局 1200 局・収集局面 13107・opening book 225 状態まで増やした版です。

- どんなデータを集めたか
- そのデータを評価関数へどう反映したか
- 探索アルゴリズムと学習情報をどう接続したか

## 学習の流れ

`npm run train-model` を実行すると、`scripts/trainModel.js` が次を行います。`--games` などの引数で規模を増やすこともできます。

1. 浅い自己対局を複数回まわす
2. 各局面から特徴量を抜き出す
3. 最終結果を教師信号にして線形重みを調整する
4. 駒の位置ごとの傾向から piece-square table を作る
5. 序盤の exact-state opening book を作る
6. `src/game/trainedModel.js` に埋め込む

## 集めている特徴量

線形モデルで使う特徴量は、次のような説明しやすいものに絞っています。

- materialDiff
- mobilityDiff
- uncertaintyDiff
- chargeDiff
- centerDiff
- kingGuardDiff
- entropy
- worldCount
- tempo
- progress
- certainDiff

「なぜこの特徴を入れたか」を README と面接で説明しやすいよう、ブラックボックスな特徴は避けています。

## piece-square table

piece-square table は、駒種ごとに「どのマスにいると最終結果が良くなりやすかったか」を owner-relative な座標で集計したものです。

- 先手はそのままの座標
- 後手は 180 度回転した owner-relative 座標

こうしておくと、先手・後手で別の表を大量に持たずに済みます。

## opening book

opening book は exact state hash をキーにしています。局面をあいまいにまとめるより、今回の規模では exact-state のまま持ったほうが説明しやすく、実装も追いやすいからです。

- 序盤数手だけを対象にする
- 各 state hash に対して action key ごとの回数と平均結果を持つ
- 少数サンプルに引っ張られすぎないよう、スコアを縮めて保存する

## learned-hybrid での使い方

学習結果は単独で使うのではなく、探索の補助として使っています。

- 序盤では opening book を優先
- 中盤以降は Expectiminimax と MCTS を両方走らせる
- 学習済み評価と action prior を合成して最終順位を決める

この構成だと、「探索」と「学習」の責務を切り分けた説明がしやすくなります。

## 限界

- 深層学習ではないので、複雑な長期依存までは拾えない
- 自己対局数は軽量で、統計的に十分とは言い切れない
- ベンチマークも低予算設定で、厳密な強さ比較ではない

その代わり、コード量に対して仕組みを把握しやすく、面接で中身をきちんと説明できることを優先しています。

## 今回の同梱設定

今回 zip に入れている学習済みモデルは、概ね次の設定で再生成しています。

- `--games 1200`
- `--epochs 40`
- `--book-depth 12`
- `--book-max-states 240`

厳密な強さ比較用というより、「軽量自己対局を実際に複数回まわし、結果をリポジトリ内に固定した」ことが分かる状態を重視しています。
