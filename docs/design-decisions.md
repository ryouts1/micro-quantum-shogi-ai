# 設計判断メモ

## 1. 量子状態は weighted worlds で持つ

「駒ごとに確率を持つ」表現より、world の集合として持ったほうが観測の意味が明確です。1 つの世界は通常の盤面として扱えるので、合法手判定や評価関数も組み立てやすくなります。

## 2. 量子移動は非捕獲に限定した

量子捕獲まで許すと、捕獲されたかどうか自体が分岐します。今回は次の見せ場を優先しました。

- 分岐の視覚化
- 観測による collapse
- AI が chance node を扱うところ

そのため、ルールの複雑さが急に跳ねる部分は切り落としています。

## 3. seed を state に持たせた

観測や MCTS が完全ランダムだと、同じ局面でも毎回違う結果になります。それでは説明しづらいので、内部 seed を state に持たせました。

この設計により、

- 観測が再現しやすい
- MCTS の比較がしやすい
- テストを書きやすい

という利点があります。

## 4. AI のランキングを UI に出した

AI を実装しても、盤面が勝手に動くだけでは説明材料が弱いです。候補手ランキング、評価値、訪問回数をそのまま出すことで、「探索が何を見ているか」を面接や README で示しやすくしました。

## 5. リプレイは state ごと保存した

着手列だけの保存よりサイズは増えますが、この作品では次の利点のほうを重視しました。

- 過去局面でもヒートマップや解析ができる
- collapse 後の実際の世界をそのまま再現できる
- サンプルデータをすぐ確認できる

## 6. UI はフレームワークを入れずに書いた

今回の主役はルールと探索です。フロントエンドの重い依存を増やすより、DOM ベースで小さく閉じたほうが次の点で有利でした。

- セットアップが軽い
- `src/game` と `src/ui` の境界が見やすい
- コード量に対して読みやすい


## 7. 学習は軽量モデルに留めた

今回の主題は、量子状態のルール設計と探索の説明です。そこで学習も、巨大なニューラルネットではなく、自己対局から作る線形重みと piece-square table に留めました。

この方針にした理由は次の通りです。

- リポジトリの中身を短時間で追いやすい
- どの特徴が評価に効くか説明しやすい
- 探索とのつながりをコード上で見せやすい

## 8. 学習結果は探索の補助として使う

学習済みモデルだけで着手を決めるのではなく、opening book、Expectiminimax、MCTS を束ねる learned-hybrid に差し込みました。

これにより、

- 序盤は book
- 中盤以降は探索
- tie-break や rollout は学習済み prior

という役割分担が明確になります。
